# Wire format

This document pins the payload encoding used by clients generated by
`protoc-gen-rust-temporal`. It is **independent** of cludden's annotation schema â€”
the annotations describe *what* methods exist; this document describes *how* the
data on the wire is encoded.

**Status:** v1, pending cross-language audit (see "Compatibility audit" below).

This file is the authoritative copy. The sibling
[`protoc-gen-ts-temporal`](https://github.com/nu-sync/protoc-gen-ts-temporal)
keeps a byte-identical mirror so the two plugins cannot drift; any change
here must be applied there in the same PR.

## Payload triple

Every workflow input, workflow output, signal payload, query payload, and update
payload uses Temporal's standard `Payload` shape with the following three
metadata + data slots filled:

| Slot                         | Value                                       |
|------------------------------|---------------------------------------------|
| `metadata.encoding`          | `"binary/protobuf"` (UTF-8 bytes)           |
| `metadata.messageType`       | `"<fully.qualified.proto.message.name>"`    |
| `data`                       | raw proto wire bytes of the message         |

### Why `binary/protobuf`?

It is the encoding used by Temporal's Go SDK proto3 payload codec. Picking it
gives us drop-in interop with workers and clients in any language that follows
the same convention â€” there is nothing custom in our framing.

### Why a `messageType` header?

Temporal's `Payload` is by design opaque to the server. To round-trip a
polymorphic value the runtime must know which proto descriptor to decode the
bytes with. The fully-qualified message name (e.g. `jobs.v1.JobInput`) is the
simplest stable key for that.

### Empty payloads

A method whose input or output is `google.protobuf.Empty` still emits the
triple â€” `messageType = "google.protobuf.Empty"`, `data = []`. We do not
substitute `null` payloads or omit the metadata. This keeps the decode path
uniform: one branch, always indexed by `messageType`.

## Decode contract

Generated code MUST:

1. Read `metadata.messageType`. If absent, **reject** the payload. We never
   fall through to a JSON default â€” silent JSON decode is a footgun that masks
   real misconfiguration (a worker emitting a payload the client can't decode
   should surface as a hard error, not as a `None` field on a typed response).
2. Look the type name up against the `TemporalProtoMessage::MESSAGE_TYPE`
   constant of the expected target type. Mismatch â†’ reject.
3. Decode `data` as that proto message's wire bytes via `prost::Message::decode`.

Generated code MUST emit:

1. The fully-qualified name of the prost message as `messageType`.
2. The wire bytes from `prost::Message::encode_to_vec(&msg)` as `data`.
3. The literal string `"binary/protobuf"` as `encoding`.

The reference implementation lives in
[`temporal-proto-runtime`](crates/temporal-proto-runtime/src/lib.rs) as
`TypedProtoMessage<T: TemporalProtoMessage>`.

## Why this lives separately from the annotation schema

Cludden's `temporal/v1/temporal.proto` schema describes annotations on rpc
methods; it does not constrain how the *runtime* serializes values. Two plugins
consuming the same proto could pick different encodings and still respect
cludden's surface. The wire format is therefore ours to specify â€” and the part
that determines cross-language interop.

## Compatibility audit

Pending cross-language test (Phase 3 in SPEC.md): verify the Go runtime shipped
by `cludden/protoc-gen-go-temporal` emits the same `(encoding, messageType,
data)` triple. If it does, a Go client written against cludden's plugin can
talk to a worker driven by `protoc-gen-rust-temporal`-generated code, and vice
versa. The TS sibling will participate in the same audit.

If the audit finds divergence, that result will be documented here along with
the choice made (match cludden vs. propose an upstream change vs. add a
compatibility-mode flag).

## Non-goals

- JSON payloads (`json/plain`, `json/protobuf`). Generated code rejects them.
- Mixed encodings within a single workflow / activity execution. One workflow
  uses `binary/protobuf` for every payload it produces or consumes.
- Compression metadata, custom encryption metadata. Out of scope; a future
  layer could wrap the triple but must preserve it intact for downstream
  decoders.
